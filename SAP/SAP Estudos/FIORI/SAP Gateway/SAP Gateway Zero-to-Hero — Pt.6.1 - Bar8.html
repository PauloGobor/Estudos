SAP Gateway Zero-to-Hero — Pt.6.1

SAP Gateway — Criando o seu primeiro serviço

Erick Carvalho
Follow
Dec 1, 2017 · 16 min read

E aweeeee meu povo!!!!!!!

Ufaaaa! Enfim, chegou o dia! Hoje iremos aprender a criar os nossos serviços utilizando o SAP Gateway.

Neste post, quero apresentar um overview sobre essa poderosa ferramenta e, pra isso, vou partir do princípio de que os componentes para o SAP Gateway já estejam instalados.

Premissas

Em um futuro próximo, posso publicar algum material de como foi a minha experiência instalando o SAP Gateway. No entanto, neste momento, partirei do pressuposto de que você já possui o componente instalado — mas segue logo abaixo algumas informações que podem lhe ajudar:

Primeiros passos…

Vamos iniciar conhecendo aquela que será a interface central usada para definir e implementar serviços. Essa interface é acessada através da transação SEGW.

Dentro do Service Builder (SEGW) veremos que ele é complementado por diversas ferramentas de suporte adicionais que auxiliam na criação e manutenção de serviços e, em certos casos, ele mesmo permite que você execute as etapas em ferramentas de terceiros para que, logo em seguida, importe os resultados.

Processo de criação de um serviço na SEGW com SAP Gateway

Neste momento vou criar um roteiro com o passo a passo da forma que foi mais simples para que eu aprendesse, e por isso neste momento vou ser o mais direto possível. Depois que você estiver mais familiarizado com as interfaces do SAP Gateway perceberá que alguns desses passos não precisam estar na sequência aqui apresentada.

O SAP Gateway é bastante versátil, e nele encontramos modos de desenvolvimento que vão desde o desenvolvedor raiz (que prefere fazer tudo na unha) até o desenvolvedor Nutella (que prefere deixar para que o framework crie a maior parte do serviço).

Eu, particularmente, utilizo no meu dia a dia as duas formas para criar meus serviços, configurando e codando a maior parte — mas às vezes também aproveito os recursos de automatização do SAP Gateway (gerando serviços com base em algo que já existe, seja uma função RFC ou uma estrutura na DDIC).

Essas duas formas de geração de serviço são:

  • Geração de serviços via SAP Gateway — Este é o modo no qual você deixa o framework criar a maior parte para você, o que no dia a dia adianta bastante o nosso trabalho (principalmente quando você está começando, é bom entender como a SAP cria seus serviços para que você tenha uma base de boas práticas). Eu conheço basicamente três modos de criação: Redefinição, RFC e por Modelo (conversaremos sobre eles no decorrer da série). Um ponto fraco sobre a geração de serviços é que fica difícil otimizá-los. Será normal a sensação de que você está engessado — esse é o preço que se paga por não ter muito esforço e velocidade na criação de um serviço.
  • Desenvolvimento de serviços — Esse é o modo faça você mesmo. Conforme já citado acima, o SAP Gateway permite que você realize a maior parte das configurações via código, tornando o desenvolvimento bastante flexível, além de torná-lo mais próximo da sua necessidade por você estar desenvolvendo apenas aquilo que você realmente precisa (ao contrário da geração de serviço que precisa prever situações genéricas, o que acaba criando mais código do que você realmente irá precisar).

Independente da solução que você planeja seguir para o desenvolvimento do seu serviço, existe um processo base que é dividido em 3 fases:

  • Modelar a definição — Esta é a primeira fase, onde você irá estruturar o seu serviço. Esta fase é extremamente importante, pois aqui é definido como consumir o serviço para que ele possa funcionar.
  • Implementar o serviço — Após a definição, inicia-se a fase de implementação: pegar todas as definições da fase anterior e fazer o serviço funcionar. Nesta fase, principalmente, optar por geração de serviço ou criação de serviço possui maior impacto de tempo de desenvolvimento. Como vimos anteriormente, dependendo da abordagem escolhida ela será mais rápida ou mais longa, mais flexível ou mais engessada;
  • Manutenção do serviço — Por último, mas não menos importante, esse é o momento em que você publicará o seu serviço, tornando-o disponível para ser consumido. Muitos blogs da SAP dizem que essa fase é feita apenas uma vez, onde após publicado o serviço não exige que você realize esta fase novamente. Na minha experiência não é bem assim, pois já tive problemas onde apenas publicando o serviço novamente consegui resolver questões relacionadas a cache.

Mãos na massa — Criando o serviço com Service Builder — SEGW

Para cumprir esse desafio, iremos criar um serviço que forneça informações sobre a nossa valiosa tabela SFLIGHT.

Criando o Projeto

Acesse a transação SEGW. Caso seja o seu primeiro acesso nesta transação, verá algo como a imagem abaixo, onde a partir do momento em que você criar os seus projetos o service builder passa a lhe apresentar todos os projetos que você vem trabalhando (isso é realmente uma opção legal do service builder, pois ele permite que você trabalhe com vários projetos em paralelo).

Clique, então, na opção de Criar (o bom e velho ícone da folhinha branca):

Após clicar em Criar Projeto, a tela abaixo será apresentada com algumas opções, como:

  • Projeto — Informar o nome do projeto. Seguindo padrão de desenvolvimento SAP, será necessário utilizar Z ou Y para desenvolvimentos que não estejam dentro de namespace ou dos domínios SAP;
  • Descrição — Descrição do projeto com o limite de 60 caracteres;
  • Tipo do Projeto — Possivelmente este campo não estará disponível para você. No entanto, esse é um recurso relativamente novo, e falaremos dele no futuro. Caso esta opção seja apresentada para você, pode deixar a primeira opção como eu fiz;
  • Estratégia de geração — Será apresentada apenas uma opção, até hoje não tive nenhuma experiência com outras opções.
  • Pacote — O bom e velho pacote, que aqui já nos mostra que, após finalizado, é possível transportar seu projeto entre ambientes;
  • Responsável — De quem é a culpa.

Criação do Modelo

Agora que temos um projeto criado, vamos para aquela que eu considero a fase mais burocrática e trabalhosa. É possível criar o modelo de algumas formas, onde as que eu já utilizei foram:

  • Por declaração — Criação de modelo na unha;
  • Baseado na DDIC — Após criação de uma base na DDIC, é possível importa-la;
  • Baseado em RFC/BOR — Similar ao modo anterior — é possível utilizar parâmetros de entrada e saída como base do nosso serviço.

Um modo que não utilizei ainda é via EDMX, mas assim que utilizar eu conto aqui sobre a experiência.

Para que possamos fixar e conhecer mais sobre as opções do service builder, iremos utilizar a opção de criação de serviço por declaração.

Clique com o botão secundário do mouse sobre o nó Data Model e, então, selecione a opção conforme a imagem abaixo:

Clicando nesta opção, será apresentada uma caixa de diálogo onde você poderá especificar o nome da sua entity (entidade). Uma observação é que o nome desta entidade é único dentro do seu projeto.

Um checkbox será apresentado para que você possa ativar a opção de criação do Entity Set (também durante este passo, eliminando assim um futuro trabalho). A criação de serviço no SAP Gateway é fortemente baseada no conceito de CRUD. Assim, você perceberá que algumas nomenclaturas fazem referência direta a conceitos de banco de dados.

Por exemplo, uma entidade nada mais é que a representação de uma unidade de dados, como uma pessoa, um lugar ou alguma coisa. Falando em ABAPês, poderíamos representar uma entidade sendo uma estrutura, enquanto que um Entity Set (coleção de entidades) seria a representação de uma tabela interna dessa entidade. Sacou?

Assim, nesta opção (criando uma entidade e setando a opção de criar o Entity Set), estaríamos criando a estrutura e sua respectiva tabela interna.

Neste exemplo não irei criar um entity set, mas caso precisasse, poderia fazê-lo posteriormente de forma bastante simples também.

Após a criação da nossa entidade, ficamos com o nosso service builder da seguinte maneira:

Temos à esquerda o nosso projeto informando o novo nó, representando a entidade recém criada e, à direita, a estrutura da nossa entidade. Não utilizaremos todos os recursos disponíveis aqui, mas posso adiantar pra vocês os mais úteis:

  • Nome — Mesmo depois de criado, o nome da entidade pode ser editado a qualquer momento;
  • Nome da estrutura ABAP — Você é capaz de informar uma estrutura da DDIC aqui, e então o SAP Gateway cria um link (herdando tamanho e características dos campos da estrutura sugerida);
  • Label — Muito útil para melhor documentação durante o consumo do serviço. Funciona como um alias da sua estrutura, porque sejamos sinceros, o desenvolvedor de outra plataforma não tem ideia do que significa VBAK, BKPF ou MSEG;
  • Media — Utilizaremos quando realizarmos upload e download de arquivos via serviço.

Seguindo. Conforme havia dito, neste momento quero criar um serviço sem base em coisa alguma, então vamos expandir o nó da entidade SFLIGHT e dar um duplo clique em Propriedades.

Nesta área, os campos obrigatórios serão: Nome, Tipo (EDM Core Type) e ao menos uma chave. Irei criar uma estrutura simples e, por isso, irei generalizar e criar o EDM como string, deixando apenas um campo como chave diferente de como foi desenhada a tabela SFLIGHT. No entanto, perceba que já irei respeitar o tamanho dos campos.

Vamos agora criar uma coleção da entidade SFLIGHT. Com o botão secundário, clique em Entity Sets > Create:

Será exibido um popup, mas seu preenchimento é bem tranquilo: basta informar o nome do Entity Set e em qual Entidade este será baseado:

Eu costumo usar como padrão de nomenclatura o seguinte formato: Nome_da_Entidade + Collection ou Nome_da_Entidade + Set (Levando em conta que utilizo o padrão Camel Case).

Neste momento, temos o modelo bem encaminhado, mas ele ainda está inativo e, além disso, é necessário que sejam geradas as classes que irão fazer o nosso serviço criar vida. Sendo assim, como ativá-los e gerar as classes base? Simples, basta clicar no botão de Generate Runtime Objects.

Após o clique você terá que informar uma request e, então, após a confirmação temos o seguinte popup que será explicado bloco por bloco, logo abaixo:

A dona SAP já propõe a nomenclatura para nós das classes bases, e eu costumo deixar. Após confirmar, será solicitado que você informe novamente qual o pacote utilizado e a request.

O que é a Model Provider Class (MPC)?

MPC é a classe que representa a definição do modelo. Logo, quando tivermos outros modelos, estes serão acrescentados à esta classe. Que uma coisa esteja clara: sempre que você alterar o modelo será necessário regerar o serviço para que essa classe seja atualizada.

O que é a Data Provider Class (DPC)?

DPC funciona em sua essência da mesma forma. Entretanto se o MPC é a classe responsável pela definição, o DPC é a classe responsável pela implementação. É no DPC que você encontrará os métodos responsáveis para executar o seu CRUD-Q (Create, Read, Update, Delete e Query).

Uma curiosidade de quem já debugou muito esse framework, a classe DPC herda a classe /IWBEP/CL_MGW_PUSH_ABS_DATA, essa é a classe base que possui os métodos para executar o CRUD-Q, além de várias outras opções de manipulação da requisição.

O que é o MPC_EXT e DPC_EXT?

Você percebeu que o framework gerou tanto para a MPC quanto para a DPC duas opções de classes? Uma chamada de classe base e a outra apenas de classe. O funcionamento disso é bem simples, as classes com sufixo DPC_EXT ou MPC_EXT (Extensão) herdam das respectivas classes bases DPC e MPC.

Isso na prática significa que caso seja necessário customizar alguma coisa dos classes DPC ou MPC (e acredite em mim, você irá customizar diversas rotinas) você deve trabalhar sempre na extensão da classe base — EXT.

Então acredito que com isso fique claro que a razão para essas extensões é permitir que nós desenvolvedores tenhamos o nosso próprio código quando necessário, e melhor que isso aconteça de forma simples apenas redefinindo ou criando novos comportamento para o método standard em sua extensão da classe, com isso não precisamos alterar a classe base. Caso você edite as classes base DPC ou MPC e então clique em regerar o serviço, perderá qualquer implementação ou alteração nesses métodos(Assim como você pode alterar as telas geradas para SM30, mas vai perder tudo se alguém regerá-las).

Para que seja possível customizar nossas extensões (MPC_EXT e DPC_EXT) iremos utilizar uma nova ferramenta, que é chamada de ABAP Class Builder. Muito interessante ter em mente, que mesmo podendo regerar os serviços a qualquer momento, o que for desenvolvido nas extensões não será reescrito ou apagado, o service builder entende que o que está escrito dentro das rotinas das extensões são customizados, e assim devem ser mantidos.

O registro do modelo e o registro do serviço é uma referencia a ativação de objetos referente ao serviço no SAP.

Implementação dos nossos serviços

Para entender o que o service builder gerou para nós, basta clicar com botão secundário sobre um dos artefatos gerados, e selecionar a opção Go To ABAP Workbench:

E você será levado para a seguinte tela:

Nesse momento você já deverá estar mais confortável, pois as ações aqui são muito similares a SE24. Perceberá também que alguns métodos estarão com a cor diferente significando que são redefinições de suas classes base.

Vamos utilizar alguns dos métodos citados acima, mas apenas fazendo uma prévia os métodos que eu costumo mais utilizar são:

  • <EntitySet>_CREATE_ENTITY — Referente ao verbo HTTP POST;
  • <EntitySet>_DELETE_ENTITY — Referente ao verbo HTTP DELETE;
  • <EntitySet>_GET_ENTITY — Referente ao verbo HTTP GET;
  • <EntitySet>_GET_ENTITYSET — Referente ao verbo HTTP GET;
  • <EntitySet>_UPDATE_ENTITY — Referente ao verbo HTTP PUT;
  • EXECUTE_ACTION — Utilizaremos bastante esse método quando optarmos por utilizar Function Import;
  • CREATE_DEEP_ENTITY — Existem situações que por exemplo, ao realizar um POST precisamos de dados de cabeçalho e de item, e ai para que não façamos duas requisições ao servidor, podemos utilizar esse recurso e mandar dados de cabeçalho e item ao mesmo tempo.

Ativação do serviço — Service Maintenance

Enfim aqui voltamos indiretamente para a SICF, essa fase nada mais é que uma automação para registrar o serviço na SICF.

Essa parte é possível fazer de duas formas, em ambientes mais atuais é possível ativar o serviço apenas dando um clique duplo em Service Maintenance, então você será apresentado a seguinte tela:

Basta clicar no botão Register e um popup requerendo algumas informações será aberto. — Não vou ir por esse caminho, pois cairemos nesse popup já já.

Optarei por realizar a configuração pela transação /IWFND/MAINT_SERVICE, isso porque algumas empresas optam por não ativar essa opção de registro do serviço pelo service builder (SEGW).

Acessando a transação acima, você será levado para a seguinte tela:

Clique no botão Inserir Serviço. Clicando nesta opção você será levado a uma nova tela, onde você irá procurar pelo seu serviço, e então inclui-lo na lista para poder realizar a ativação na SICF.

Informe o nome do serviço e aperte enter

Selecione a linha do serviço que você deseja registrar, e então clique na opção Inserir serviços selecionados. Com a popup aberta, você terá poucas opções, entre elas estão, alterar o nome do serviço técnico e do modelo, informar o pacote e se deseja já criar um nó na ICF de forma automática. Sendo sincero sigo esse padrão nos meus desenvolvimentos, e chegando aqui apenas confirmo a criação.

Após o okay você pode voltar para a tela anterior (/IWFND/MAINT_SERVICE) e então pesquisar novamente pelo seu serviço clicando na opção Filtro. E um novo popup será apresentado:

Alias do sistema

Como vimos sempre que utilizamos uma função para poder basear o nosso serviço, ela necessitará ser uma função RFC. Isso porque o componente do SAP Gateway pode, e ao meu ver deve, estar separado do SAP ECC. Assim, para que seja possível a comunicação entre sistemas será necessário um mapeamento que poderá ser visualizado na SM59 como na imagem abaixo:

Tendo o mapeamento feito é preciso ter uma configuração na:

SPRO > SAP NetWeaver > SAP Gateway > OData Channel > User Settings > SAP Gateway to SAP System > Manage SAP System Aliases

Essa configuração é a responsável por criar os alias dos nossos sistemas, assim quando necessitarmos informar o Aliás com qual sistema SAP ECC o nosso SAP Gateway deve se comunicar, assim ele irá buscar nessa configuração.

Testando o serviço

Sim a SAP possui a sua própria ferramenta para testar serviços, e ela pode ser acessada via transação /IWFND/GW_CLIENT.

Existe bastante praticidade ao utilizar essa ferramenta, principalmente pois informando o nome do serviço na URI, clicando no botão EntitySets é possível escolher de forma bem prática qual serviço você deseja testar.

Entretanto, eu ainda prefiro o Postman para essa atividade, e por isso nos meus exemplos irei utiliza-lo como referência para testar os meus serviço.
